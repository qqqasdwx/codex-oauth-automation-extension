# 本仓库与上游仓库差别

本文档只说明**面向用户可感知的功能差异**，不展开代码实现细节。

如果你已经在用上游仓库，可以把这个仓库理解为：**在尽量跟随上游主流程的基础上，额外强化了手机号验证、邮箱池接入、重复跑号稳定性，以及 Release 下载体验。**

## 目前和上游相比，主要多了什么

### 1）新增 `Hero-SMS` 手机号验证能力

本仓库支持在流程进入 `add-phone` 时，接入 `Hero-SMS` 完成手机号验证。

用户侧能感知到的差异：

- 可以在侧边栏直接配置 `Hero-SMS`
- 支持进入手机号验证分支后继续往下跑，不用完全手工接管
- 对“号码迟迟收不到验证码”“同一号码多次接码后失效”这类情况，做了更积极的超时更换与重试处理

这部分更适合本来就需要手机号验证、或者经常卡在 `add-phone` 的用户。

### 2）新增 `OutlookEmail（邮箱池）` 模式

本仓库支持把已部署的 `OutlookEmail` 服务接入到扩展里，作为新的邮箱来源。

接入这个能力的主要原因是：**上游仓库原本偏向 Hotmail 账号池方案，但在一些实际使用场景里，Hotmail 账号池维护成本偏高、API 调用也容易频繁失败，而且有些用户并不希望本地额外启动 helper。**

因此，这个仓库额外接入了 `OutlookEmail`，用于覆盖下面这类场景：

- 不想维护 Hotmail 账号池
- 不想在本地单独启动 helper
- 希望直接通过已部署的 Web 服务 / API 管理邮箱池
- 希望把“待注册邮箱”和“注册成功邮箱”按分组管理

项目地址：

- https://github.com/assast/outlookEmail

用户侧能感知到的差异：

- 可以在侧边栏填写 `OutlookEmail Base URL` 和登录密码
- 可以拉取分组，并选择“注册邮箱池分组 / 注册成功分组”
- 流程会自动从源分组分配邮箱来跑注册
- 注册成功后，会自动把当前邮箱移动到成功分组
- 移动完成后，侧边栏分组数量会自动刷新

如果你本来就在维护邮箱池，这个功能会比手动填邮箱更省事。

### 3）重复跑号场景更偏向“清状态再开始”

本仓库对重复执行流程时的页面残留问题做了额外处理。

用户侧能感知到的差异：

- 新一轮开始前，会更主动地清理旧页面状态
- 更适合连续多轮自动跑
- 能减少上一轮页面残留、旧标签页、旧会话对新一轮的干扰

### 4）验证码提交与分支判断做了额外稳定性处理

本仓库针对验证码提交阶段做了一些增强，重点是减少误判和无效提交。

用户侧能感知到的差异：

- 更偏向“确认页面状态稳定后再继续”
- 对“已注册账号”和“新注册账号”的分支判断更谨慎
- 在部分容易出现跳页、误判、空提交的场景下更稳一些

### 5）Release 页面额外提供“可直接使用的扩展打包 zip”

本仓库的 GitHub Release，除了 GitHub 默认生成的 `Source code (zip)` 之外，还会额外提供一个**面向使用者的发布包 zip**。

用户侧能感知到的差异：

- 可以直接下载扩展发布包
- 包内只保留扩展运行所需文件
- 不会把测试、参考仓库、工作流等开发文件一起塞进去

如果你只是想下载安装扩展，这个包会比 GitHub 默认的源码包更直接。

## 目前没有刻意改动的部分

- 主体 10 步自动化流程仍然尽量跟随上游
- 上游已经合入的能力，这里原则上也会继续跟进
- 这个仓库的目标不是完全另起一套，而是做**偏实战场景的增强 fork**

## 适合什么用户

如果你符合下面任意一种情况，这个仓库通常会比上游更适合你：

- 你需要接 `Hero-SMS` 做手机号验证
- 你已经有 `OutlookEmail` 邮箱池，希望直接接入扩展
- 你经常是多轮自动跑，希望每轮开始更稳一点
- 你更希望直接下载可用发布包，而不是 GitHub 默认源码包

如果你只想使用最原始、最接近上游的版本，那上游仓库会更合适。
